# 更新日志生成器

从 git 提交记录生成结构化的更新日志。重点是把技术性提交转换为用户能理解的描述。

## 执行流程

### 1. 确定范围

先确认用户想要的时间或版本范围：

```bash
# 查看最近的 tag
git tag --sort=-creatordate | head -10

# 查看最近的提交概览
git log --oneline -20
```

根据用户请求选择范围：
- "自上次发布以来" -> `git log <last-tag>..HEAD`
- "过去一周" -> `git log --since="1 week ago"`
- "v1.0 到 v2.0" -> `git log v1.0..v2.0`
- 未指定 -> 默认从最近的 tag 到 HEAD

### 2. 获取提交记录

```bash
git log <range> --format="%H|%s|%b|%an|%ai" --no-merges
```

如果仓库使用 Conventional Commits（大多数情况下是的，尤其是配合 git-commit skill 使用时），可以直接按 type 前缀分类。

### 3. 分类变更

将提交归入以下类别，按重要性排序输出：

| 类别 | 对应 type | 说明 |
|------|----------|------|
| 新功能 | feat | 用户可感知的新能力 |
| 改进 | perf, refactor（影响用户体验的） | 现有功能的增强 |
| 修复 | fix | 问题修复 |
| 破坏性变更 | 含 BREAKING CHANGE 的提交 | 不兼容的变更，需要特别标注 |

**过滤掉**以下类型（用户不关心）：
- `chore`, `ci`, `build`, `test`, `style` -> 内部维护，不进入更新日志
- 纯重构（不影响用户体验的 `refactor`）-> 跳过
- 合并提交 -> 已通过 `--no-merges` 过滤

### 4. 转换语言

将开发者视角的提交信息转换为用户视角的描述：

**转换前** (开发者语言):
```
feat(api): 添加批量导出接口
fix(auth): 修复令牌刷新后的重定向问题
perf(db): 优化查询缓存策略
```

**转换后** (用户语言):
```
- 支持批量导出数据
- 修复了登录后偶尔需要重新登录的问题
- 数据加载速度显著提升
```

转换原则：
- 去掉 type/scope 前缀，用自然语言描述
- 聚焦用户能感受到的变化，而非技术实现
- 合并相关的小改动为一条描述
- 如果提交信息本身已经是用户语言，保持原样即可

### 5. 输出格式

```markdown
# 更新日志 - <版本号或日期>

## 新功能

- **功能名称**: 简要描述用户能做什么

## 改进

- 改进点描述

## 修复

- 修复了什么问题

## 破坏性变更

- 变更描述及迁移说明
```

格式规则：
- 不使用 emoji 前缀
- 每条描述一行，用 `-` 开头
- 重要功能可以加粗标题
- 破坏性变更必须说明迁移方式

### 6. 写入文件

默认追加到项目根目录的 `CHANGELOG.md` 顶部（在已有内容之前）。如果文件不存在则创建。

写入前向用户确认：
- 展示生成的更新日志内容
- 确认写入位置和格式

## 特殊场景

### 提交信息不规范时

如果仓库没有使用 Conventional Commits，通过 diff 内容推断分类：
- 新文件/新函数 -> 新功能
- 修改现有逻辑 -> 改进或修复（根据提交信息判断）
- 删除代码 -> 可能是清理，跳过

### 空范围

如果指定范围内没有提交，告知用户而不是生成空文档。

### 与 git-commit skill 的协作

如果仓库的提交已经通过 git-commit skill 规范化，分类工作会非常简单——直接利用 type 前缀即可，重点放在语言转换上。
