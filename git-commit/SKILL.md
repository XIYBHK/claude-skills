# Git 提交 Skill

目标是把“提交代码”这件事做得更稳：先看清变更，再决定是否拆分，最后生成规范提交信息并执行提交。

## 何时使用

遇到下面这些请求时，直接使用此 skill：

- 用户要求提交当前代码改动
- 用户要求生成 commit message 或 Conventional Commits 提交信息
- 用户要求分析 git status / git diff 后帮忙提交
- 用户要求把多类改动拆成多个提交
- 用户已经写了一部分提交信息，希望你补全、规范化或代为执行提交

如果用户只是讨论 git 概念、解释 commit 规范，或者只想查看状态而不提交，就不必强行执行完整提交流程。

## 核心原则

1. **先分析，再提交**：先看 `git status`、`git diff`、`git diff --staged` 和最近提交历史，再决定怎么提交。
2. **提交信息反映意图**：提交信息要体现“为什么改”，不要只罗列文件名。
3. **优先保持提交干净**：如果发现不相关改动混在一起，先建议拆分，而不是把所有内容硬塞进一个提交。
4. **遵守仓库现有风格**：最近提交历史比抽象规范更重要，必要时优先贴合当前仓库习惯。
5. **采用安全的 git 流程**：不要跳过 hooks，不要擅自 amend 旧提交，不要用 `git add .` 或 `git add -A` 一把梭，优先按文件精确暂存。
6. **只在用户明确要提交时提交**：如果用户只是让你”生成提交信息”或”看看怎么提交”，那就只给建议，不执行 `git commit`。
7. **不遗漏任何文件**：提交前必须列出所有变更和未跟踪文件，确保每个文件都被分配到某个提交批次。如果某个文件确实不该提交（如临时文件、构建产物），要明确告知用户并说明原因，而不是默默跳过。

## 提交前检查流程

### 1. 审查工作区

并行执行以下 git 检查：

- `git status --short` 或 `git status`
- `git diff --staged`
- `git diff`
- `git log --oneline -5`

目标：

- 看清哪些文件已暂存、哪些还未暂存、哪些是未跟踪文件
- 理解改动的真实内容，而不是只看文件名
- 学习仓库最近的提交风格、scope 命名和语言习惯

如果变更很多，不要直接假设是一次提交；先按主题归类。

### 2. 判断是否需要拆分提交

出现以下情况时，优先建议分批提交：

- 同时包含 `feat` 和 `fix`
- 文档、重构、构建配置、业务逻辑混在一起
- 多个互不相关的目录或模块同时变更
- 自动生成文件与手写代码混在一起

如果需要拆分：

- 先给出一个简短的提交计划
- 说明每个批次大概包含哪些文件、对应什么 type/scope
- 得到用户认可后再逐批提交

如果改动本来就是一个完整功能或一个单一修复，就保持单次提交，不要过度拆分。

## 提交信息规范

遵循 Conventional Commits，默认使用中文描述：

```text
<type>(<scope>): <简短描述>
```

必要时可扩展为多行：

```text
<type>(<scope>): <简短描述>

- 变更点 1
- 变更点 2
```

### 常见 type

- `feat`：新增功能
- `fix`：修复问题
- `refactor`：重构或结构优化，但不改变外部行为
- `docs`：文档更新
- `test`：测试相关
- `perf`：性能优化
- `build`：构建流程、依赖、打包系统
- `ci`：CI/CD 配置
- `style`：纯格式或风格调整
- `chore`：杂项维护
- `revert`：回滚提交

### scope 选择

- 优先使用仓库里已经存在的模块名、包名、目录名或功能域
- 单一模块用单个 scope，如 `auth`、`api`、`skill-creator`
- 如果仓库通常不写 scope，可以省略
- 只有在确实表达更清晰时，才使用多个 scope

### 描述写法

- 用中文、简洁、偏祈使或结果导向表达
- 聚焦改动目的，不要只写“更新文件”
- 避免空泛表述，如“修改代码”“优化内容”

示例：

- `feat(api): 添加批量导出接口`
- `fix(auth): 修复令牌刷新后的重定向问题`
- `docs: 补充技能触发与提交流程说明`

更具体的风格参考见 `references/commit-examples.md`。

## 执行提交的工作方式

### 场景 A：用户要你直接提交

按下面顺序进行：

1. 审查变更与最近提交风格
2. 判断是单次提交还是分批提交
3. 生成提交信息草案并向用户说明你的判断
4. 用精确文件列表执行 `git add <file...>`
5. 执行 `git commit`
6. 再次运行 `git status` 确认结果

如果提交被 hook 拦下：

- 不要绕过 hook
- 先阅读报错并修复问题
- 重新暂存相关文件
- 创建一个**新的**提交，不要默认 amend 旧提交

### 场景 B：用户只要提交信息

只需输出建议的 type、scope 和候选 commit message，不要执行 `git add` 或 `git commit`。

**候选数量硬上限：恰好 2-3 条，绝不超过 3 条。** 原因：给太多选项反而增加选择负担，2-3 条足以覆盖不同风格偏好。如果你觉得只有 1 条最优，就只给 1 条，但永远不要给第 4 条。

输出格式参考：

```text
推荐 type: <type>
推荐 scope: <scope>

候选提交信息：
1. <推荐写法> — <一句话说明适合什么情况>
2. <备选写法> — <一句话说明适合什么情况>
3. <备选写法>（可选）
```

如果改动混杂（如 feat + fix），只需一句话说明建议拆分及各部分的 type/scope，不需要为每个拆分后的提交写出完整 message。

## 分批提交策略

当用户同意拆分提交后：

1. 先列出批次计划
2. 每个批次只包含一个清晰主题
3. 每批提交前再次确认该批文件集合是完整的
4. 每次提交后保留剩余改动，继续处理下一批

输出示例：

```text
建议拆成 3 个提交：
1. docs(git-commit): 重写技能触发与提交流程说明
2. refactor(skill-creator): 清理评测脚本组织方式
3. chore: 删除废弃参考文件
```

拆分时重点关注：

- 是否把同一功能的测试、类型、配置遗漏在工作区里
- 是否把自动生成文件混入错误批次
- 是否有未跟踪文件应该加入当前批次

## 风险与边界

- 不要提交疑似凭证、密钥、`.env` 等敏感文件；如果用户明确要求，先提醒风险
- 不要擅自 push、创建 tag、创建 PR；这些是提交之后的独立动作
- 不要为了“整理历史”擅自改写已有提交
- 不要因为用户说“顺手一起提交”就把无关改动打包进去

## 输出建议

### 单次提交成功后

简要汇报：

- 提交 hash
- 最终提交信息
- 工作区是否已清理干净

### 提议阶段

如果还没执行提交，优先给出：

- 你对本次改动的判断
- 推荐的提交信息
- 是否建议拆分，以及理由
